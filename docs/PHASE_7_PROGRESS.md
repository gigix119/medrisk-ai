## Architecture decisions for Phase 7

1. **Two new packages, mirroring the existing `medrisk_ml`/`medrisk_inference` split:**
   - `app/research/` ‚Äî DB-facing, **light** (no numpy/torch/sklearn). Domain enums, Pydantic schemas, repositories (SQLAlchemy queries), and services that only need Postgres + stdlib (`hashlib`, `json`): dataset quality audit, leakage audit (exact-overlap only), study-config validation, artifact-manifest hashing. **Importable by the live API** ‚Äî `app/api/v1/endpoints/research.py` uses these directly.
   - `medrisk_research/` (new top-level package, sibling to `medrisk_ml`/`medrisk_inference`) ‚Äî the offline CLI bridge: `validate-study`/`load-study`/`quality-audit`/`leakage-audit`/`ingest-evaluation`. **Revised mid-session**: this package's actual commands read already-computed artifact files (a completed `medrisk_ml` experiment's `metrics.json`/predictions CSV) and write to Postgres - they never call `medrisk_ml.evaluation.*` directly, so they need only Postgres + PyYAML + stdlib, not numpy/sklearn/torch. PyYAML was added to `requirements-dev.txt` only (never `requirements.txt`/the production image) so this package's tests can run against the real Postgres test database in `tests/integration/`, the same tier as everything else - not a separate `tests/ml`-style heavy-deps tier as originally planned. A *future* `evaluate` command that runs fresh inference and recomputes metrics would need `requirements-ml.txt` too, in an environment that also has Postgres access - no current CI job has both; flagged under "Remaining tasks". Never imported by `app/main.py` or any `app/api/*` router either way, so the live API's import graph stays untouched.
2. **`POST /api/v1/research/evaluations` creates a `pending` row only.** The live API process has no sklearn/numpy (see dependency tiers above), so it cannot itself run a *fresh* evaluation. This mirrors the existing precedent that training/evaluation only ever happens via CLI (`medrisk_ml.cli train/evaluate`), never via API request. The API's job is to read back what the CLI already computed and persisted.
3. **Reuse existing tables instead of duplicating them**, per the master prompt's own instruction ("extend only where necessary"):
   - `Dataset` (Phase 6) is already the dataset-*version*-grain entity (`slug` + `version` unique) ‚Äî no separate `DatasetVersion` table. Added one nullable `manifest_hash` column instead.
   - `DatasetSample` (Phase 6) is already the manifest row-grain entity ‚Äî reused as the FK target for `EvaluationSamplePrediction`, not duplicated.
   - `ModelDeployment` (Phase 3) is already the model-identity/registry table for served models ‚Äî reused as an optional enrichment FK on `EvaluationRun`/`ExperimentRun` (nullable, since registry artifacts like `smoke-baseline-cnn:0.0.1-smoke` were never necessarily loaded as a live deployment).
4. **Six new tables** (one Alembic migration): `research_studies`, `dataset_quality_audits`, `dataset_leakage_audits`, `experiment_runs`, `evaluation_runs`, `evaluation_sample_predictions`. Enum columns follow the existing `SAEnum(..., values_callable=...)` convention (see `ModelDeploymentStatus`/`PredictionStatus`).
5. **Scientific integrity policy is centralized** in `app/research/domain/enums.py` (the classification enums themselves, which structurally prevent e.g. `scientific_maturity="clinically_validated"` from ever being a valid value) and `app/research/domain/policy.py` (the disclaimer text + small validation helpers, e.g. rejecting a study config that asks to calibrate/tune-threshold on the `test` split).
6. **Honest partial linkage, never fabricated:** `EvaluationSamplePrediction.dataset_sample_id` is nullable; `EvaluationRun.notes` records the exact count of unresolved sample-key links for the ingested smoke experiment (22 of 32, per the subset-size mismatch above) rather than silently dropping or fabricating them.
7. **Metrics are never recomputed by the live API.** `medrisk_ml.evaluation.metrics.compute_binary_metrics` (already tested, already correct re: NaN-for-undefined) is the one source of truth; `medrisk_research`'s ingestion CLI calls it (or reads its already-written `metrics.json`) and persists the result as JSON. The API only ever reads `EvaluationRun.metrics`/`.confidence_intervals`/etc. back out ‚Äî it never imports `medrisk_ml`.
8. **CLI location:** new package `medrisk_research/cli.py`, not `app/cli/*` as the master prompt's illustrative naming suggested ‚Äî chosen because the existing repo convention for heavy/offline entry points is a top-level sibling package with its own `cli.py` (`medrisk_ml/cli.py`, `medrisk_inference/cli.py`), not a `cli` subpackage inside `app/`. Documented here so a future session doesn't go looking for `app/cli/`.
9. **Frontend routes are `/app/research` and `/app/research/:evaluationId`**, not the master prompt's illustrative `/research/results/:evaluationId` ‚Äî kept inside the existing `/app/*` authenticated tree (mirrors every other authenticated page; there is no separate top-level `/research` layout) and named by resource (`evaluationId`) rather than by the word "results", matching `routes.predictionDetail(id)`'s existing naming style. Added as a secondary nav entry (desktop sidebar bottom block + mobile "More" dialog, labelled "Evaluations") rather than a primary `appSidebarItems`/bottom-nav slot, both because the bottom nav is already at its documented 4-item cap and because this mirrors the existing precedent for "Active model"/Help/Preferences ‚Äî secondary, not daily-use, navigation.
10. **`ResultClassificationBadge` deliberately renders every `ResultClassification` value in one identical neutral style** (no green/red/gold color-coding by perceived rigor). Color-coding would itself imply some classifications are "better" or more trustworthy than others, which is exactly the kind of implicit validity ranking the scientific-integrity policy forbids ‚Äî the badge's only job is to disclose which evaluation protocol produced a number, never to grade it.
11. **The result page only fetches metrics/confusion-matrix/error-rows when `EvaluationRun.status === "completed"`.** A `pending`/`running`/`failed`/`cancelled`/`invalidated` run has no persisted metrics to show (`EvaluationRun.metrics` is `NULL` until `mark_evaluation_run_completed` runs) ‚Äî the page renders a plain "no completed metrics yet" message instead of firing three requests that would just return empty/zeroed shapes.
12. **Frontend types are generated, not hand-written**, via the existing `npm run api:generate` (`scripts/export-openapi.mjs` dumps `app.main.app.openapi()` straight from Python, then `openapi-typescript` regenerates `frontend/src/api/generated/schema.d.ts`) ‚Äî re-run this session specifically to pick up the six new Phase 7 schemas/endpoints. `frontend/src/features/research/api.ts` only re-exports/narrows those generated types; it does not redeclare any response shape by hand.
