# ADR-001: Backend architecture for Phase 1

- **Status**: Accepted
- **Date**: 2026-06-24
- **Context**: MedRisk AI Phase 1 — backend foundation, no ML yet.

## Decision

Build the backend as a Python modular monolith: **FastAPI** + **async SQLAlchemy 2.0** + **PostgreSQL** + **Alembic** + **PyJWT** + **pwdlib (Argon2)**, with explicit repository/service layering and integration tests run against a real PostgreSQL database (never SQLite, never mocks).

## Considered alternatives

### Web framework: FastAPI vs. Django vs. Flask

- **Django** brings its own ORM, admin, and forms system. Its synchronous-first ORM (Django's async ORM support is comparatively new and partial) fights the goal of a fully async data path, and its admin/forms machinery isn't needed here.
- **Flask** is minimal and unopinionated, which is appealing, but means hand-building what FastAPI provides out of the box: request validation, OpenAPI generation, and a first-class async story.
- **FastAPI** was chosen for native async support, Pydantic-based validation that doubles as OpenAPI documentation, and a dependency-injection system that makes per-request resources (like a DB session) explicit and testable.

### Architecture: modular monolith vs. microservices

A Phase 1 backend with three tables and four feature areas (health, auth, users, predictions) does not have the scale or team-boundary pressure that justifies microservices' operational cost (multiple deployments, network calls where there used to be function calls, distributed transactions). A modular monolith — clear internal layers, one deployable — gets the organizational benefit (you can't accidentally call the database from an HTTP handler) without the operational cost. If a genuine scaling reason emerges later (e.g., the ML inference workload needs independent scaling/GPU nodes), splitting that one module out remains straightforward because it's already isolated behind `app/services/prediction.py`.

### Database access: async SQLAlchemy 2.0 vs. sync SQLAlchemy vs. an ORM-less query builder vs. an alternative ORM

- **Sync SQLAlchemy** under an async framework means either blocking the event loop or running every query in a thread pool — extra complexity for no benefit when an async driver (`asyncpg`) and async SQLAlchemy both exist and are stable.
- **A query builder without an ORM** (raw SQL + a thin wrapper) would reduce "magic" but means hand-writing every mapping between rows and Python objects, and losing typed relationships — a poor trade for a project whose explicit goal is to also be a learning vehicle for professional backend patterns.
- **SQLAlchemy 2.0's typed, `Mapped`/`mapped_column` style** was chosen specifically because it is the *current*, actively maintained idiom (not the legacy 1.x `Column`/`declarative_base()` style), gives real static type checking via mypy, and its async support (`AsyncSession`, `async_sessionmaker`) is first-class rather than bolted on.

### Database: PostgreSQL vs. SQLite vs. MySQL

SQLite cannot represent this project's actual target (a real deployed Postgres-backed API) and would hide real concerns — connection pooling, native `ENUM`/`JSONB`/`INET` types, concurrent-write behavior — that the project explicitly wants to practice. PostgreSQL 16 was chosen for its strong typing (native enums, `JSONB`), maturity, and because it's the overwhelmingly common choice for this kind of workload in industry, making the skills transferable. MySQL would work technically but offers no advantage here and weaker native JSON/enum ergonomics.

### Migrations: Alembic vs. `Base.metadata.create_all()`

`create_all()` is fine for a quick prototype but has no concept of *changing* a schema once it has data in it, and no downgrade path. Alembic was chosen because schema evolution — adding a column, changing a constraint — is a certainty for any project with more than one phase, and because reviewing an autogenerated migration before applying it is a valuable habit to practice early (this project's own initial migration needed a manual fix: native Postgres `ENUM` types aren't dropped automatically on `downgrade()` and had to be cleaned up explicitly — exactly the kind of issue blind trust in autogenerate would have missed).

### Auth tokens: PyJWT vs. python-jose; passwords: pwdlib/Argon2 vs. passlib/bcrypt

- **`python-jose`** is effectively unmaintained and has had multiple disclosed vulnerabilities; **PyJWT** is the actively maintained, narrowly-scoped alternative and was chosen for that reason alone.
- **`passlib`** is also unmaintained at this point (no released support for newer Python/cryptography combinations going forward) and defaults many users toward bcrypt, which has a 72-byte input truncation footgun. **`pwdlib`** is its actively maintained spiritual successor; **Argon2** (the winner of the Password Hashing Competition) was chosen over bcrypt/PBKDF2 as the hashing algorithm itself, configured explicitly (`PasswordHash([Argon2Hasher()])`) rather than relying on a "recommended default" that could change between library versions.

### Repository/service separation vs. "fat" endpoint functions

Putting database queries directly in endpoint functions is faster to write initially but makes business rules ("a duplicate email is a conflict," "an inactive user can't log in") untestable without spinning up HTTP, and makes the same rule easy to implement inconsistently if it's needed from two different endpoints. A thin repository layer (no HTTP concepts) under a service layer (the actual business rules, given an `AsyncSession` and `Settings` explicitly rather than hidden globals) keeps each piece independently testable without introducing speculative abstract base classes or a "framework within the framework."

### Testing: real PostgreSQL integration tests vs. SQLite-for-tests vs. fully mocked

SQLite-for-tests is a common shortcut, but it silently passes tests that would fail against real PostgreSQL — different `ENUM` handling, different JSON behavior, different constraint enforcement timing. Since this project explicitly leans on Postgres-native features (enums, `JSONB`, `INET`), testing against SQLite would test a different, friendlier database than the one actually deployed. A second, dedicated `medrisk_test` database alongside the development database was chosen instead — real PostgreSQL, full fidelity, but never touching development data, and isolated per test via post-test `TRUNCATE`.

## Consequences

- Running tests requires a real, reachable PostgreSQL instance (native install, or Docker) — there is no "tests just work with zero setup" path. This is treated as worth the fidelity it buys.
- Adopting SQLAlchemy 2.0's typed style and Alembic's async template means following patterns that are less ubiquitously documented in older tutorials than the legacy 1.x style; the trade-off is correctness and forward-compatibility.
- The modular-monolith boundary between `services/prediction.py` and the rest of the app is where a future ML-serving split, if ever needed, would happen — this was a deliberate seam, not an accident.
